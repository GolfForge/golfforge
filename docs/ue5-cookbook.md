# UE5 / MCP / PCG cookbook

The engine-side knowledge base: hard-won pitfalls, working recipes, and the current Fab/Megascans asset list. Read on demand when building courses or working on the sim. **Append new gotchas/recipes here** as you hit them — `CLAUDE.md` links here and stays slim.

## UMG procedural layout: full-width bottom bar (GOL-145)

Mixed canvas anchors (stretch one axis, point the other) are fiddly to get right for a full-bleed bottom bar. The robust recipe: a **full-screen `UVerticalBox`** as the root canvas's only child (`SetAnchors(FAnchors(0,0,1,1))` + `SetOffsets(FMargin(0))`), then:
- a **fill `USpacer`** first (`UVerticalBoxSlot::SetSize(FSlateChildSize(ESlateSizeRule::Fill))`) — pushes everything to the bottom;
- bottom-left content with `VerticalBoxSlot::SetHorizontalAlignment(HAlign_Left)` (sizes to content);
- the **full-width bar** last with `SetHorizontalAlignment(HAlign_Fill)` — the border stretches edge-to-edge, no anchor math.

Hit-testing: set the `UVerticalBox` (and the fill spacer) to `SelfHitTestInvisible` / `HitTestInvisible` so the empty middle passes clicks through to gameplay; only the actual control widgets (bar, cards, dropdowns) capture input. (`FSlateChildSize`/`ESlateSizeRule` live in `Components/SlateWrapperTypes.h` — include it.)

For autosize **point-anchored** panels, **only `Offsets.Left` (X) and `Offsets.Top` (Y) position the widget** — `Offsets.Right/Bottom` are the width/height, which autosize computes and therefore *ignores*. Alignment then shifts the widget by `-Alignment * size`. So to inset a bottom/right-aligned panel **away from that edge you must use a negative Left/Top**, NOT Right/Bottom. Concretely, a bottom-centre card (`anchors (0.5,1)`, `align (0.5,1)`, autosize): `Offsets.Top = 0` sits its bottom flush on the screen edge; `Offsets.Top = -110` lifts it 110px up. (Corrected GOL-146: the earlier "bottom-center uses Offsets.Bottom" note was wrong — setting `Bottom` did nothing across three rebuilds; negative `Top` is what moved it.)

Heads-up: moving a panel to the bottom can collide with other bottom-anchored widgets — the swing meter's bottom-centre card overlapped the GOL-145 control bar. Z-order doesn't help if both are visible; reposition one (GOL-146 lifted the meter with a negative `Offset.Top` per above).

## UMG `FSlateRoundedBoxBrush`: radius 999 fails to render on wide boxes (GOL-146)

`RoundedBrush(Col, 999.f, ...)` (the "fully rounded pill" idiom from CSS `border-radius: 999px`) renders fine on small **square** widgets (e.g. `MakeStatusDot`, ~9x9) but **silently doesn't draw** on wide/short rectangles (a 404x15 progress track, a header pill). The fill/outline just vanish — easy to misread as "my widget isn't being added" or "the color's too faint". Use a concrete radius instead: `Radius::Sm` (8) for pills, or ~half the box height (`7.f` for a 15px track) for pill ends. The rest of the UI already does this (the GOL-145 status pill uses `Radius::Sm`, `MakeKbd` uses 6) — 999 was the outlier. Symptom checklist when a themed box is invisible: (1) radius too large, (2) for canvas children, a zero `Offsets.Right` = zero width.

## UMG buttons floating over the 3D scene need a fill, not a ghost (GOL-147)

`MakeGhostButton` (transparent fill + hairline border, hover lifts to a faint surface) reads well **on a panel/card** but looks orphaned floating directly over the gameplay viewport — the user flagged the range Menu button as "transparent, doesn't look like other buttons with a darker bg". Fix: give scene-floating buttons a **filled glass chip** — `StyleButton(Btn, Color::GlassFill(), Radius::Sm, Color::Border(), 1.f)` + a `Body` label. Same control reads fine as a ghost when it sits on a glass panel (the in-round round-panel Menu button stays ghost). Rule: pick ghost vs filled by what's *behind* the button, not by what it does.

## Modal opened from another modal: close the parent first (GOL-147)

The leave confirm can be reached from inside the Settings modal (Settings → "Main Menu"). Rather than stack modals (z-order + focus + double input-gating headaches), `RequestLeaveToMainMenu()` calls `CloseSettings()` first, then shows the confirm over the bare HUD — so a Cancel returns to the HUD, not back into Settings. On dismiss, `CloseLeaveDialog()` hands keyboard focus back with `FSlateApplication::Get().SetAllUserFocusToGameViewport()` (the same idiom `CloseSettings` uses) so Space/gameplay keys work again. Each modal still owns its keys while up via `SetIsFocusable(true)` + `SetKeyboardFocus()` on open and an `NativeOnKeyDown` Esc/Enter handler.

## Assets loaded via LoadObject path strings won't cook unless always-cooked (GOL-148)

`GolfUITheme` resolves its fonts + gradient materials with `LoadObject("/Game/UI/Fonts/...")` /
`"/Game/UI/Materials/..."` path strings (so widgets compile/render before the assets exist, and so the
theme has no hard asset dependency). The cooker builds its set from **static references** — it can't
see a runtime path-string load, so those assets are silently dropped from a packaged build (you'd get
the engine-default font fallback + flat-colour gradients in the cooked exe, but everything works in the
editor). Fix: add the directory to always-cook in `Config/DefaultGame.ini`:
`+DirectoriesToAlwaysCook=(Path="/Game/UI")` under `[/Script/UnrealEd.ProjectPackagingSettings]`.
General rule: any `/Game/...` asset reached only by a runtime string (LoadObject/StaticLoadObject,
soft paths built at runtime, console-driven loads) needs an always-cook directory or an explicit hard ref.

## Marketplace / Fab asset dependencies

**License basis (open-source compliance, GOL-44):** Quixel Megascans / Megaplant and marketplace foliage are licensed under the **Fab Standard License**, which lets you ship them *baked into a product* but forbids redistributing the raw assets "on a standalone basis" — so committing the `.uasset` files to this public repo is not allowed. We therefore **gitignore all of them** and each machine re-downloads them from its own Fab account. Nothing here is committed, so there's no redistribution and no attribution obligation in the repo. (Only assets offered under **CC-BY/CC0** on Fab could be committed — CC-BY with attribution in `NOTICE` — but the packs below are Standard-License, not CC.)

Gitignored: `engine/**/Content/Megaplant_Library/`, `Megascans/`, `Fab/`, and `PC3D_Kentucky_Bluegrass_v14/`.

**UE5.7:** Quixel Bridge is deprecated; Megascans now comes via the Fab plugin (Window → Fab) and imports under `/Game/Fab/...`; some marketplace packs import to their own top-level `/Game/<Pack>/` (gitignore those explicitly). ("Asset not available in UAsset format" = use the Fab plugin's Add-to-project, not the old Bridge.)

**Fresh clone — fetch these from Fab before opening `PracticeRange` / `GolfForgeDemoBlack`** (else you'll see "missing asset" warnings):
- `Tree_Silver_Birch` (Megaplant) → `/Game/Megaplant_Library/...` — range/course trees (PCG scatter).
- Fab Megascans surfaces `Lawn_Grass_tkynejer` / `Uncut_Grass_oilpt20` / `Clover_Patches_on_Grass_sgmkajak` / `Bright_Desert_Sand_sjzkfega` / `Concrete_Floor_virrebs` → `/Game/Fab/...` — ground materials.
- `PC3D_Kentucky_Bluegrass_v14` → `/Game/PC3D_Kentucky_Bluegrass_v14/...` — fairway 3D grass.

(`Thatching_Grass` / `Ribbon_Grass` / `Kikuyu_Grass` / `025_Grass` were trialed but the build references KBG — no need to fetch them.)

**Contributor-availability note:** Each machine fetches these from its own Fab account. Under the Fab Standard License the **Personal tier is free** below the revenue threshold (~$100k lifetime), so most contributors get the free-listed assets at no cost; a couple (`PC3D_Kentucky_Bluegrass_v14`, and some Megascans that went paid in 2025) may need to be purchased. ## Course-import landscape path (UE5.7)

The primary all-Python landscape-import path via `unreal.LandscapeEditorObject.import_landscape_data()` is **dead in UE5.7** — `LandscapeEditorObject`, `LandscapeProxy.Import`, `LandscapeSubsystem`, and `WorldPartitionEditorSubsystem` are all unbound in the Python API. Use the manual fallback: human clicks Landscape mode → Manage tab → New Landscape → Import from File. Python automation handles everything else (material creation via `MaterialEditingLibrary`, WP cell loading via `WorldPartitionBlueprintLibrary.load_actors`, sphere spawn + physics setup, screenshots). For repeatable course imports later, either (a) write a stdlib-only PNG decoder + the Texture2D → RenderTarget → `landscape_import_heightmap_from_render_target` chain, or (b) escalate to a paid MCP that wraps the editor's import flow.

## PCG Python API reference

Verified across `build_pcg_treescatter.py` (graph) + `scatter_full_landscape.py` (volume/generate/perf) — those two scripts are the source of truth.

- `graph.add_node_of_type(SettingsClass) -> (PCGNode, PCGSettings)`; `graph.add_edge(from_node, Name(from_pin_label), to_node, Name(to_pin_label))`; `graph.remove_node`/`remove_edge`; `graph.get_input_node()`/`get_output_node()`; `graph.get_editor_property("nodes")`.
- Pin labels: `pin.get_editor_property("properties").get_editor_property("label")`; pins via `node.input_pins`/`node.output_pins` (properties, not methods). Input node OUT label=`In`; Output node IN label=`Out`; SurfaceSampler IN=`Surface`,`Bounding Shape`; AttributeFiltering kept-pin=`InsideFilter`.
- Modern class names: Create-Attribute = **`PCGAddAttributeSettings`** (PCGCreateAttributeSettings is deprecated); point filter = **`PCGAttributeFilteringSettings`** (PCGPointFilterSettings deprecated). Surface needs **`PCGGetLandscapeSettings`** (`sampling_properties.get_layer_weights=True`) on the SurfaceSampler `Surface` pin AND the graph **Input** node on the `Bounding Shape` pin — SurfaceSampler yields 0 points without a bounding shape.
- **Reusing the scatter graph for a 2nd map at a different density: give it its OWN graph.** Density lives on the SurfaceSampler node *inside the shared graph*. GolfForgeDemo is validated at 0.02 ppsm; bumping the shared graph for the range (0.35) would mean a later GolfForgeDemo scatter that fires *before* rebuilding the graph places ~17x the trees (perf disaster). `build_pcg_treescatter.py` now reads `GRAPH_PATH`/`DENSITY_PPSM` from `globals()` and seeds a missing sibling via `EditorAssetLibrary.duplicate_asset(BASE, GRAPH_PATH)`, so `DENSITY_PPSM=0.35; GRAPH_PATH="/Game/PCG/PCG_TreeScatter_Range"; exec(...)` authors a denser range graph in one call. `scatter_range_trees.py` is the range orchestrator — **persistent** (`generation_trigger=GenerateOnLoad`, save the umap, regenerates each load), the opposite of `scatter_full_landscape.py`'s transient measure-only gate.
- **InstancedSkinnedMeshComponent instance introspection (PCG SkinnedMeshSpawner output).** Count = `len(comp.get_editor_property("instance_data"))` — `get_instance_count()`/`get_num_instances()` silently return 0 for *skinned* ISM. Per-instance transform = `idata[i].get_editor_property("transform")` → a `Transform3f` whose `Vector3f` components are OPAQUE to Python (`repr` is `{}`, `to_tuple()` returns `()`, no `.x`); the reliable read is `t.export_text()` + regex on `Translation=(X=..,Y=..)`. `get_instance_transform(i, True)` also doesn't work on skinned ISM.
- **Live weightmap re-import regenerates PCG.** Re-importing a landscape's weightmaps (Landscape ▸ Manage ▸ Import, Heightmap unchecked) triggers an automatic PCG regen, so trees relocate to the new paint on import — re-running the orchestrator just confirms/forces it.
- **The GPU hogs are geometry/skinning, not pixels: 3D landscape grass + skinned PCG trees.** On the range, 4K fullscreen tanked to ~45 FPS. `stat unit` showed GPU-bound (36 ms vs ~13 ms game), and *lowering* `r.ScreenPercentage` made it **worse** — a tell that the cost is vertex/skinning, not fill (so TSR/DLSS upscaling won't save you). Diagnose by elimination: `grass.Enable 0` (live), toggle trees, watch the GPU ms delta. The dense camera-driven KBG `LandscapeGrass` on the fairway was ~half the GPU; trees the rest. Megaplant trees are **skeletal (skinned) meshes** — no static `SM_` variant ships — so each instance pays per-frame skinning regardless of resolution; many close + on-screen (a tight tree corridor) is expensive. Fixes that worked: remove the grass output from the range material, and keep tree density modest (range runs 0.10 ppsm ≈ ~1,900). A static-Nanite tree import would remove the skinning cost entirely.

## Known pitfalls / pending issues

- **Mac packaged build — borderless locks at the macOS "Looks like" logical resolution, not the panel's physical pixels** (GOL-72, deferred). On a 4K monitor where macOS Display preferences are set to the default "Looks like 1920×1080" (common for readability), borderless mode locks the resolution dropdown at 1920×1080 because UE asks the OS for the desktop res and gets the logical value back. **Workaround for users: use Fullscreen mode** (the resolution dropdown enumerates the panel's actual modes including 4K; UE drives exclusive fullscreen directly). Or change System Settings → Displays → "More Space" so macOS reports the panel's native res, then borderless follows. Mac perf today is TSR-only (no DLSS/XeSS on Apple, MetalFX not available in this UE5.7 install — GOL-64 blocked), so at native 4K expect the M4 base to land ~25-30 FPS with TSR Performance + low quality settings. The "proper" engine-side fix is engine-source territory (NSScreen backing-pixels vs logical points) and not worth the time until MetalFX is unblocked.
- **`execute_unreal_python` sets `__file__` to a throwaway temp script, not the real script path.** It's `engine/Golfsim/Intermediate/UnrealClaudeMCPPython/exec_*.py`. Any `engine/scripts/*.py` that derives a repo path from `__file__` (e.g. `build_range_splatmap.py`'s `OUT_DIR = <root>/courses/...`) silently writes to the WRONG place (`engine/Golfsim/courses/...`) when exec'd in-editor vs run standalone. Guard: trust `__file__` only when `os.path.basename(dirname(__file__)) == "scripts"`, else hardcode the repo root.
- **Don't nuke+recreate a referenced material (e.g. a landscape material).** `build_*_material.py`'s backup→delete→recreate pattern fails on a re-run: `delete_asset` on a material the landscape references is blocked, so `create_asset` at the same path returns `None` (the script then `AttributeError`s) AND the failed delete **nulls the landscape's `landscape_material` → the terrain renders the default checker**. Recover by re-binding (`landscape.set_editor_property("landscape_material", mat)`). To *change* a live referenced material, edit it in place — e.g. to drop 3D grass, delete the `Landscape Grass Output` node in the Material Editor by hand (range-only; the course material keeps its grass). Note: you can't enumerate a material's expressions from Python in 5.7 (`editor_only_data` doesn't expose the collection, `Material.expressions` is protected, and `MaterialEditingLibrary` has only `get_num_material_expressions` / `delete_all` / delete-by-object — no get-all), so surgical node removal in script isn't available; the manual editor delete is the path.
- **Mac Cowork sandbox can't `unlink` files in `.git/`.** Symptom: `.git/index.lock` accumulates. Fix from host Mac terminal: `rm -f .git/index.lock`.
- **Mac Cowork sandbox is Linux, not macOS.** Don't create `.venv` from the Cowork bash — it'll have Linux-ELF binaries that the host Mac can't execute. Use `./setup.sh` from the host Mac terminal.
- **Windows Cowork sandbox can't `git clone` into the workspace mount.** Same family as the Mac note: writing into `.git/config` of a new clone fails with `Operation not permitted`. Do clones from PowerShell on the Windows side.
- **Sandbox `git status` shows LFS-tracked PNGs as "modified" on every fresh session.** The Linux Cowork sandbox doesn't ship `git-lfs`, so it can't run the smudge/clean filter and sees pointer-vs-binary diffs. Ignore in the sandbox; on Windows with LFS installed, `git status` is clean.
- **`.gitattributes` has a latent ordering bug.** `*  text=auto eol=lf` catch-all comes AFTER the `*.png ... -text` LFS lines, and git applies the LAST matching attribute, so `text=auto` wins on PNGs. Not breaking anything (LFS still works, content survives), but worth fixing: move the catch-all to the top of the file, or scope it to text extensions.
- **MSVC 14.51 (VS2026) is "not a preferred version" per UE5.7.4.** UE wants 14.44. Soft warning during build; the project compiled fine. If we hit weird linker errors on plugin builds later, this is the first suspect.
- **UI fonts are embedded in the `.uasset`, not loaded from the `.ttf` (GOL-138).** Barlow / Barlow Condensed / JetBrains Mono live at `engine/Golfsim/Content/UI/Fonts/{Barlow,BarlowCondensed,JetBrainsMono}.uasset` (+ weighted `FontFace`s under `Faces/`), imported with `FontLoadingPolicy::LazyLoad` — the font bytes are stored *inside the package* (asset size ≈ the source `.ttf`; no `.ufont` sidecar), so the assets are self-contained and cross-machine safe. Other machines (incl. the **Mac cook**) just pull the `.uasset` via LFS and use it — they **never re-import**. The raw `.ttf`s in that folder are only a reimport source + the OFL-1.1 license copy, so UE's "N source content files changed — import?" prompt on them is harmless: **decline it** (reimport only re-dirties identical assets). If fonts ever fall back to Roboto on another machine or in a cook, the suspects are (a) the font `.uasset` didn't smudge from LFS (pointer left in place), or (b) `GolfUITheme`'s getter paths (`/Game/UI/Fonts/{Barlow,BarlowCondensed,JetBrainsMono}`) drifted from the asset names — **not** a missing `.ttf`. The composite weight→face wiring was authored via `CompositeFont.import_text` because `FFontData`/`FTypefaceEntry`/`FTypeface` aren't exposed to Python (struct-text round-trip from an engine font is the only scriptable path).
- **UMG: a content-sizing widget (notably `UButton`) wrapping a panel shrinks it to its content — even inside a Fill slot (GOL-139).** Symptom: cards/tiles collapse to their *text* size and float, no matter what the outer layout does (`UGridPanel` fill OR `HBox/VBox` `FillWidth/FillHeight` both fail identically). A `UButton` (Slate `SButton`) places its content at the content's *desired* size; the fill slot stretches the (transparent) button but not its content, so the visible card never grows. **Fix:** root the interactive widget in a **layout panel** (`UOverlay`/`UBorder`) — which fills its slot and stretches `Fill`-aligned children — and handle click/hover via `UUserWidget` overrides (`NativeOnMouseButtonDown` / `NativeOnMouseEnter` / `NativeOnMouseLeave` + `SetVisibility(Visible)` so the tile is hit-testable) instead of a wrapping button. For a CSS-grid-like **responsive** layout, place cells with a **`UCanvasPanel` + proportional anchors** (`FAnchors(minX,minY,maxX,maxY)` as fractions; with min≠max, `SetOffsets` margins become *insets* → gutters) — anchored slots fill their rect by definition and scale with the window (× the DPI curve in Project Settings → User Interface). That's the reliable bento/grid recipe; `UGridPanel` fill is unreliable. Related: Slate has **no native linear/radial gradient brush** (`UBorder` is solid/9-slice) — gradients (tile scrims, bg ambiance, hover-wash) need a small UI material `M_UIGradient` (tracked in GOL-150); flat fills stand in until then.
- **UMG segmented/HBox-of-buttons: Fill-equal slots clip the longest label (GOL-142).** A row of option buttons each set to `FSlateChildSize(ESlateSizeRule::Fill)` get *equal* width = trackWidth/N, so a long label ("Everyone holes out") next to a short one ("Gimmes on") gets squeezed to the short one's share and clips. Fix = size each option to its **content** (just `AddChildToHorizontalBox(B)` with no `SetSize`) so the track hugs the labels — which also matches the prototype's `.seg button { padding: 9px 18px }` (content-sized, no flex-grow). Applied in `USegmentedControl::Rebuild`; also gained optional dim **sub-labels** (a second `Mono(9)` text in the button's HBox, recolored to `AccentInk` with selection) and a **per-option disable** API (`SetOptionDisabled`) for "one live option among disabled siblings".
- **In-game HUD overlay = a full-screen `SelfHitTestInvisible` widget; read level actors find-only (GOL-144).** A HUD widget that floats panels over the live viewport (round panel + hole map) uses a full-screen `UCanvasPanel` root set to `ESlateVisibility::SelfHitTestInvisible` in `NativeOnInitialized` — the root passes clicks through to gameplay/other widgets while its own buttons (Menu) still hit-test. Mount it between the legacy panel (z0) and the modals (z30+), e.g. `AddToViewport(20)`, and drive it per-frame from `Tick` (cheap reads of `URoundSubsystem::GetState()`; to-pin = `Dist(pawn, pin XY)/CmPerYd` live). **Reading the env director (`AGolfRangeEnvironment`) for sky/time must be find-only** (`TActorIterator`, no spawn) — the existing `GetOrSpawnRangeEnv` *spawns + applies* an env on demand, which would re-light the **course** map mid-round. Mode split: in a round show the round HUD and `Collapsed` the legacy range panel (club still works via Q/E); on the range do the reverse.
- **Three procedural-UMG footguns hit while building the wizard player row (GOL-143).** (1) `using namespace GolfUI;` brings in free functions `Body(int)`/`Display(int)`/`Mono(int)` — a local `UHorizontalBox* Body = ...` then **shadows** `GolfUI::Body`, so `Body(15)` becomes "call a `UHorizontalBox*`" → `C2064 term does not evaluate to a function`. Name layout locals `BodyRow`/`BodyPad`, not `Body`. (2) `GolfUI::StyleButton` sets an 18px label `NormalPadding`; in a small fixed icon button (e.g. a 30×30 handicap −/+ stepper) that padding is wider than the button and shoves the glyph off-centre — build a tight `FButtonStyle` with `SetNormalPadding(0)` for icon buttons instead. (3) a `UBorder`'s content alignment defaults to **Fill**, so a single centered glyph (an avatar initial) renders top-left and reads "dim/wrong" — call `Border->SetHorizontalAlignment(HAlign_Center)` + `SetVerticalAlignment(VAlign_Center)`.
- **Round config rides in `FRoundState`, not the world subsystem (GOL-142).** To make a wizard-time choice (hole subset, gimme radius) affect gameplay that runs on a *different* map after an `OpenLevel`, put it in `FRoundConfig` and store it on `FRoundState` — `URoundSubsystem` is a **GameInstanceSubsystem** (survives map travel), and world-scoped consumers like `URoundHoleOutSubsystem` read it via `URoundSubsystem::Get(this)->GetState()`. Don't try to push state onto the world subsystem before the load (it's recreated on the new world). The state machine (`GolfsimRound::StartRound`/`OnHoleHoled`) is schedule-driven, so "play holes 1–9" = pass a 9-element schedule (subset via the pure, unit-tested `SelectHoles`); `round.complete` fires after the last entry for free. Defaulted struct params on `StartRound` (both the subsystem and the pure namespace) keep existing callers/tests untouched.
- **UMG: some widgets won't repaint a `SetStyle`/`SetBrush`/visibility change in certain containers — unsolved (GOL-141 → GOL-153).** Building the round-setup wizard's header stepper, the active-step pill **would not paint** by any brush mechanism: `UButton::SetStyle(FButtonStyle)` per state (the exact pattern `USettingsMenu`'s rail uses successfully elsewhere), a `UBorder` background nested in the pill content via `SetBrush(RoundedBrush(...))`, and pre-built base+accent layers toggled by `SetVisibility` — **all three failed to repaint** for these widgets sitting in the centered topbar `UHorizontalBox` (brand | spacer | stepper | spacer | close-X). Meanwhile, in the *same* `RefreshStepper()` call, `UTextBlock::SetColorAndOpacity`/`SetFont` **did** repaint, and the *identical* rounded-box brushes paint fine on `UCourseCard` and on the close-X / Continue buttons in the same widget. Net: the active step is conveyed by **text only** (bright-green/bold number + white/bold name) and the design's filled-circle badge was dropped. Root cause not found — suspect a layout/sizing or invalidation quirk specific to auto-sized buttons/borders between `USpacer`-filled siblings. If you hit a "style won't repaint" wall like this, the proven escape hatch is to build the visual as a **standalone `UUserWidget`** (like `UMenuTile`/`UCourseCard`, which paint reliably) rather than inline buttons/borders. Tracked in GOL-153.
- **Gradients in UMG = a UI material; rounded gradients = a per-tile-sized SDF mask (GOL-150).** `engine/scripts/build_ui_gradient_materials.py` builds two `MD_UI` + `BLEND_TRANSLUCENT` materials (`M_UIGradient{Linear,Radial}` under `/Game/UI/Materials/`) used as `UImage` brushes via `GolfUI::MakeLinearGradient` / `MakeRadialGradient` (each widget makes its own MID; flat-colour fallback if the asset is missing). A flat rectangular gradient over a **rounded** card shows square corners — fix is a rounded-box SDF mask baked into the linear material via a `MaterialExpressionCustom` node (`float2 p=abs(UV*Size-Size*0.5)-(Size*0.5-Radius); float d=length(max(p,0))+min(max(p.x,p.y),0)-Radius; return saturate(0.5-d);`) multiplied into Opacity, with `SizeX/SizeY` (px) fed per-tile from `GetCachedGeometry().GetLocalSize()` and `Radius` matching the card's `FSlateRoundedBoxBrush`. The crisp rounded *card* still uses `FSlateRoundedBoxBrush` (cheap); the material mask is only for the gradient overlay. Build via `MaterialEditingLibrary` (mirror `build_golf_green_material.py`); the Custom node wires up with `unreal.CustomInput()` + `CustomMaterialOutputType.CMOT_FLOAT1`, and connects work (no "connect failed" warnings) — but see the next two bullets.
- **`AssetTools.create_asset` fails ("create_asset failed", returns None) while the editor is in a play session (GOL-150).** Asset create/delete (and the nuke+recreate `build_*_material.py` pattern) are blocked during PIE/standalone; the log shows `LogUtils: The Editor is currently in a play mode`. Exit **all** play sessions — in-viewport PIE *and* the separate standalone-game window — before running a material builder via `execute_unreal_python`. (Diagnose with `unreal.get_editor_subsystem(unreal.UnrealEditorSubsystem).get_game_world() is not None`.) Re-running is otherwise safe for these UI materials — nothing references them as a hard asset dependency (widgets `LoadObject` them at runtime), so the cookbook's "don't nuke a referenced material" caveat doesn't apply here.
- **The editor silently rewrites `Golfsim.uproject` — `git diff` it before every commit (GOL-150).** Observed: the editor stripped the `"SupportedTargetPlatforms": ["Win64"]` guard off the **DLSS / Streamline / XeSS** plugin entries. That guard keeps those NVIDIA-Windows-only plugins out of the **Mac** cook, so committing the rewrite would break the Mac build. Revert it (`git checkout -- engine/Golfsim/Golfsim.uproject`) unless the change is intentional. (The uproject also trips the CRLF→LF `.gitattributes` warning — harmless.)
- **Cowork mode runs inside Claude Desktop.** Quitting Claude Desktop (e.g., to reload the MCP config) ends the active Cowork session. Always update `CLAUDE.md` and commit before restarting Claude Desktop so the next session picks up cleanly.
- **Overpass mirrors are flaky.** Script now retries across 4 mirrors. If all fail, just try again later — usually a few hours fixes it. (Observed: main `overpass-api.de` was 504ing all evening on May 13; `kumi.systems` recovered by morning.)

## Course atmosphere = a shared placed `AGolfEnvironment` director + editor "Apply To Level" (GOL-161)

The range's runtime sky director was generalized from `AGolfRangeEnvironment` to a course-agnostic **`AGolfEnvironment`** (`engine/Golfsim/Source/Golfsim/GolfEnvironment.{h,cpp}`) so one system drives every map: the range PIE-spawns it (interactive Time x Sky dropdowns), the course **places one in the umap** at a fixed preset. Reusable for future pipeline courses + GOL-154 live weather. (No CoreRedirect needed for the rename — the actor was only ever PIE-spawned, never serialized in a umap.)

- **Editor-time stamp for bake-readiness.** A runtime BeginPlay-only director does NOT help Lightmass (GOL-101) — the bake reads the *placed actors' authored values*. So the director exposes `UFUNCTION(CallInEditor) ApplyToLevel()` (`#if WITH_EDITOR`) that writes sun/fog/skylight (+ find-or-spawns an unbound `APostProcessVolume`) into the level; save the umap to persist. It deliberately **skips the cloud MID** — a transient `UMaterialInstanceDynamic` must never be assigned to a *placed* VolumetricCloud (unsaveable on the umap); clouds are runtime-only and aren't baked anyway. `ApplyToLevel` IS Python-callable: `actor.call_method("ApplyToLevel")`.
- **`Cloud_GlobalCoverage` is a BIAS, not a 0-1 fraction — and higher = THINNER.** `m_SimpleVolumetricCloud_Inst` defaults: coverage **-0.2**, density **0.008** — and that default is a pleasant moderate sky. Setting coverage 0.5 made it *sparse*. The range's old R4 Sky presets (Clear 0.05 / Cloudy 0.45 / Overcast 0.85, density 0.30-0.70) were authored on a 0-1 "higher = more clouds" model, so they render inverted + far too dense; left as-is (range ships this way, a remap is a follow-up). The course's `FewClouds` preset just uses the material defaults (-0.2 / 0.008).
- **Canonical "tournament Saturday afternoon" course preset** (code defaults in `GolfEnvironment.cpp`, applied via the placed director in `GolfForgeDemoBlack.umap`): DirectionalLight **Movable**, pitch -35 (~35deg above horizon), 5000K, intensity 8; ExponentialHeightFog 0.018; SkyLight **Movable + RealTimeCapture**, scale 1.1; PostProcessVolume **unbound**, auto-exposure locked (min=max=1.0) bias 1.0, bloom 0.5, vignette 0.4, color grade cool shadows (0.95,0.98,1.05) / warm highlights (1.05,1.0,0.95); FewClouds sky. Cloud cost is absorbed by DLSS (`golfsim.SetUpscaler 1`) at the perf gate.
- **Editor Python property-name gotchas.** A `bFoo` UPROPERTY drops the `b`: `bApplyPostProcess` -> `apply_post_process`. Component mobility is set on `actor.root_component.set_editor_property("mobility", unreal.ComponentMobility.MOVABLE)`, NOT `actor.set_mobility()`. The cloud component material reads via `cc.get_editor_property("material")` (no `get_material()`); read a MIC scalar with `MaterialEditingLibrary.get_material_instance_scalar_parameter_value(mat, "Cloud_GlobalCoverage")`.
- **Live tuning:** `golfsim.SetTimeOfDay <0-24>` (`#if WITH_EDITOR`) drives the active director's sun by a day-arc; `golfsim.SetTime`/`SetSky <idx>` step presets.
- **Night is deferred (GOL-169).** Dark-sky + raised-ambient reads either unplayably black or like daytime; Night needs artificial lights + a night grade. Commented out of `TimePresets` (seeds preserved in the constructor) so the dropdown is Dawn/Morning/Noon/Dusk/Afternoon.

## 3D fairway grass — deferred to GOL-170; grass-map gotcha + the Grass Detail toggle backend (GOL-162)

> **Status: 3D grass rendering is DEFERRED to GOL-170.** The toggle backend below shipped and stays live (dormant, Settings UI hidden); the material ships grass-free. The notes here are the reference for the GOL-170 investigation.

**Why it never rendered (the decisive UE 5.7 gotcha):** `grass.GrassMap.UseRuntimeGeneration` defaults to **0**, so LandscapeGrass spawns **only from pre-built/serialized grass density maps** (Build → Build Grass Maps). With it off and no maps built, you get **0 instances even with the grass output correctly wired** — silent. The two fixes: (a) Build Grass Maps (baked, serialized into the umap), or (b) enable runtime generation (`grass.GrassMap.UseRuntimeGeneration=1` + `grass.GrassMap.AlwaysBuildRuntimeGenerationResources=1`) — on-the-fly, no bake, smaller umap (best for the data-driven course pipeline). Diagnose with: count HISM instances under the Landscape (0 = not spawning), then check this cvar before suspecting the material.

**Material re-wire is script-only:** `MaterialEditingLibrary` (5.7 Python) can't enumerate expressions — only `get_num_material_expressions` (count) + `delete_all_material_expressions`. So you can't surgically reconnect/dedupe a grass node; the canonical path is re-running `build_course_material.py` (backup→nuke→recreate). A material can hold **only one** Landscape Grass node — adding a second (because you couldn't see/remove the disconnected first) fails to compile ("The material can contain only one Landscape Grass node") and the whole material falls back to default. Recovery: `git checkout` the `.uasset` + `EditorLoadingAndSavingUtils.reload_packages([pkg])`.

The course fairway grass was **already wired and committed** before GOL-162 (the ticket's "never wired" was wrong), then hand-detached for dev perf. `engine/scripts/build_course_material.py:431` calls `_build_grass_output(mat)` **unconditionally** (Fairway layer only, `grass=True`), binding `LGT_FairwayGrass` (5 Kentucky-Bluegrass Nanite varieties, density 10000, cull **10–35 m** by deliberate perf choice). The range ships grass **off** (`build_range_material.py` `BUILD_GRASS=False`, GOL-5). So GOL-162 was a *toggle + verify*, not a wiring task.

- **The toggle = a user setting driving engine cvars, not a material edit.** `FGolfDisplaySettings.GrassDetailLevel` (0=Off 1=Low 2=High) → `grass.densityScale` + `grass.CullDistanceScale` (both confirmed-present writable floats via `find_console_variables grass.`; maps Off→(0,0) Low→(0.5,0.7) High→(1,1)). **Off = the painted KBG-green Fairway texture as the standalone fallback** — verify the fairway still reads with grass off. Persists in a custom `[GolfForge.Graphics] GrassDetail` ini key (the pin-distance GConfig idiom, not a scalability level), applied in `GolfDisplay::Apply` + at boot.
- **Custom display cvars need a boot-time apply.** `GolfDisplay::Apply` only runs from the settings menu / console, so a persisted custom cvar (grass, and pre-existingly DLSS) does NOT re-apply on a cold launch. Fix: `FGolfsimModule::StartupModule` registers `FCoreDelegates::OnPostEngineInit` → `GolfDisplay::ApplyGrassDetailFromSaved()` (grass-only; sets the two cvars from the saved/`#if PLATFORM_MAC` default). This is also how the **Mac-conservative default (Low)** takes effect before any menu opens.
- **Nanite + this grass: already the good case.** The KBG meshes are **Nanite-enabled with OPAQUE materials** (`MI_Grass_Blades*`, `BLEND_OPAQUE` — verified via `inspect`/Python: `mesh.nanite_settings.enabled=True`). `r.Nanite.Foliage=True` is set project-wide (`DefaultEngine.ini`). Opaque geometry blades are exactly what Nanite wants — no 2× masked-material penalty, no WPO conflict — so LandscapeGrass spawning Nanite-opaque meshes is the efficient path; no grass change warranted. Nanite Foliage's *experimental, tree-focused* big wins (assembly/voxel/skinning, "500k tree instances") apply to **trees → GOL-167 / GOL-31**, not grass (Epic + community guidance: small grass often matches/beats Nanite as traditional instances; masked grass cards cost ~2×).
