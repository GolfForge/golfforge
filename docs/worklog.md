## 2026-05-23 — Practice Range + range shot tool (Windows)

- **Practice Range level** at `/Game/Maps/PracticeRange.umap` (Basic level), now the **default map** (`DefaultEngine.ini` `EditorStartupMap`/`GameDefaultMap`). Flat 504 m walkable launch-monitor test range: centered 400×100 yd fairway strip, tee box at the −X end, rough fill, perimeter tree band. Three reusable scripts: `build_range_splatmap.py` (stdlib-only PNG writer → 4 mutually-exclusive masks), `build_range_material.py` (`M_PracticeRange`, fork of the course material — same Fab surfaces + KBG tint + Fairway grass output), `build_range_lighting.py` (sun/fog/skylight). PlayerStart at the tee facing +X, `BP_FirstPersonGameMode` so WASD walks. Verified in PIE: driver shot carries ~241 m down the fairway with tracer + landing marker. (Manual editor steps, 5.7 Python-unbound: new Basic level, flat New Landscape 8×8 @ section 63 / scale 100, Paint reusing existing LIIs, Manage>Import the 4 splats.)
- **Range spacebar shot tool**: `AGolfRangeHUD` (set as the GameMode HUDClass). In PIE, number keys 1-6 select a club from a preset bag (Trackman PGA-Tour averages), Space hits a randomized shot (per-shot dispersion on speed/launch/spin/azimuth), selected club + last carry render bottom-right. Reuses the C++ solver + `AGolfBallActor`. Removed the Space→IA_Jump mapping so hitting doesn't jump.
- **Range R1 — tee-fixed arrow-key aiming** (backlog §R1): on the range the player is planted on the tee and Left/Right arrows yaw the view + aim. All in `GolfRangeHUD.h/.cpp`: a constructor enables `Tick`, which integrates yaw into the control rotation (the FP camera follows); `EnsureInputBound` calls `SetIgnoreMoveInput`/`SetIgnoreLookInput` on the controller (range-only — only this HUD does it, so BethPage walking is untouched) and binds Left/Right press+release on the legacy `InputComponent`; `FireRandom` now reads `PC->GetControlRotation()` for the shot heading so view = shot. No shared-asset (IMC/BP) edits. Verified in PIE: WASD + mouse-look dead, aim swept −34.9°→+27.8° across shots, carries correct per club.
- **Range R2 — UMG metrics panel + club dropdown** (backlog §R2): replaced the canvas HUD with a **pure-C++ `UUserWidget`** (`GolfRangePanel.h/.cpp`) — the project's first UMG widget, **no WBP asset**. Builds its whole tree in `NativeOnInitialized` (CanvasPanel → top-right anchored Border → VerticalBox: title, a 6-row label/value GridPanel, a `UComboBoxString`). Top-right grid refreshes per shot: Club · Ball Speed (mph) · Launch (deg) · Spin (rpm, backspin) · Carry (yd) · Offline (yd L/R), SI→display converted in `FireRandom`. The dropdown's 6 bag clubs set the active club via an `OnClubChosen` `TFunction` back to `AGolfRangeHUD::SelectClub`; 1-6 keys stay shortcuts and sync the dropdown (`SetSelectedClubIndex`, feedback-guarded against the `ESelectInfo::Direct` re-broadcast). `AGolfRangeHUD` creates + `AddToViewport`s the panel in `EnsureInputBound`, and `DrawHUD` is gutted. Two interaction fixes found in PIE: (1) the FP session captured the mouse → range now sets `bShowMouseCursor` + `FInputModeGameAndUI` (range-only, alongside the R1 input lock) so the dropdown is clickable; (2) a picked combobox kept keyboard focus so Space toggled it → after a pick the widget defers `SetAllUserFocusToGameViewport` a tick so Space fires. Dropdown row text set to white (`ItemStyle.TextColor`) since the default is dark. `Golfsim.Build.cs` now links `UMG`/`Slate`/`SlateCore`. Verified in PIE: panel renders, dropdown + 1-6 stay in sync, Space fires after a pick, grid updates in yd/mph (e.g. Driver ~167 mph / ~265 yd). The new-UCLASS + module-dep build was editor-closed; the three interaction tweaks were body-only and went in via Live Coding. A post-feature code-review pass (3 agents) hardened it: weak-ptr captures (HUD callback + deferred focus timer), club names fed from `GBag` (no duplicate list), empty-options guard, dead include removed.
- **Range cursor — themed golf-tee hardware cursor** (QOL, bundled with R2): the white-background source art (`assets/images/main_cursor_full.png`, a glowing golden tee) was matted to transparency offline with .NET `System.Drawing` (flood-fill the connected white from the edges + a brightness-ramp alpha so the glow stays a soft semi-transparent halo, not a pale ring), rotated 180° to point up-left, and downscaled to a 64px `engine/Golfsim/Content/Slate/golf_tee_cursor.png`. The range HUD sets it via `World->GetGameViewport()->SetHardwareCursor(EMouseCursor::Default/Hand, "Slate/golf_tee_cursor", hotspot)` in the same range-only controls-lock block (hotspot ~ (0.063, 0.047) on the sharp tip). Verified at 4K fullscreen. Size for non-4K screens is a later TODO (PNG swap, no rebuild — hotspot is normalized).
- **Range R3 — perimeter trees, reshaped into a corridor frame** (backlog §R3): trees framing the range via the existing PCG path, **no graph topology change**. The range landscape already had a painted `Trees` weight band, so this was orchestration. Key call: the range gets its **own** graph `/Game/PCG/PCG_TreeScatter_Range` at its own density rather than sharing BethPage's (validated at 0.02) — a separate graph avoids the shared-graph density footgun (a later BethPage scatter would otherwise inherit the range's value and badly over-place trees). Authored by making `build_pcg_treescatter.py`'s `GRAPH_PATH`/`DENSITY_PPSM` overridable via `globals()` + seed-a-missing-sibling by duplicating the base graph (additive, BethPage defaults unchanged). New `scatter_range_trees.py` orchestrator — a fork of `scatter_full_landscape.py` with the **opposite** intent: persistent (`generation_trigger=GenerateOnLoad` + save the umap) vs BethPage's transient measure-only gate; volume at origin ±25300 cm. First scatter put a tree ring on the full 504 m perimeter (10,808 trees), but from the tee that read as a vast empty plain (side tree lines ~252 m out). Reworked `build_range_splatmap.py`'s layout from a perimeter ring into a tight **400×70-yd open lane** framed by a 22-yd tree wall (trees ±35 yd off the centerline; fairway narrowed 100→50 yd); land past the wall stays rough, hidden behind the trees. Regenerated the 4 splats → USER re-import (which auto-regenerates the PCG) → re-scatter, verified ~99% in the corridor frame / open lane clear by parsing instance `export_text`. Also fixed `build_range_splatmap.py`'s `__file__` root detection (exec'd in-editor it was writing splats to `engine/Golfsim/courses/` instead of the repo `courses/`). **Perf pass — 4K fullscreen had tanked to ~45 FPS.** Added a `stat fps` + render-resolution readout to `AGolfRangeHUD`. `stat unit` showed GPU-bound (36 ms vs ~13 ms game) and *lowering* screen percentage made it **worse** → geometry/skinning-bound, not fill. Toggling trees off saved only ~8 ms; the real hog was the **3D fairway grass** (dense camera-driven KBG via `M_PracticeRange`'s `LandscapeGrassOutput`, ~half the GPU). Removed it range-only by **deleting the grass-output node** in the Material Editor — rebuilding the material via script is unsafe (the landscape references it, so `build_range_material.py`'s nuke+recreate aborted on the blocked delete and nulled the landscape binding → checker; re-bound to recover). `build_range_material.py` gained a `BUILD_GRASS=False` default so a future clean rebuild omits grass; `M_GolfsimCourse` keeps its grass. Trees also read as a solid "wall of green" up close, so density was dialed 0.35 → **0.10 ppsm (~1,966 trees)**. Result: **~50 FPS / 16 ms GPU** at 4K. Persisted via `PracticeRange.umap` save (GenerateOnLoad). NOTE: Megaplant trees are skeletal (skinned) meshes — no static variant ships — so they stay relatively costly up close; importing a static-Nanite tree later would buy more headroom.
